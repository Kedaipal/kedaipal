# Internationalisation (i18n)

Kedaipal ships in **English (`en`, base) and Bahasa Malaysia (`ms`)** using
[Paraglide JS](https://inlang.com/m/gerre34r/library-inlang-paraglideJs) (inlang).

## How it works

- **Source of truth:** `messages/en.json` and `messages/ms.json` — flat
  `key → string` catalogs. `en` is the base locale.
- **Compiled output:** the Paraglide Vite plugin (`vite.config.ts`) compiles the
  JSON into typed message functions under `src/paraglide/` on every `dev`/`build`.
  This directory is generated — never edit it by hand. To recompile manually:
  ```sh
  npx @inlang/paraglide-js compile --project ./project.inlang --outdir ./src/paraglide
  ```
- **Usage:** `import { m } from "../paraglide/messages"` then call
  `m.key_name()` in JSX. Keys with placeholders take an object:
  `m.cost_cta_join({ price: 104 })` for `"Become a Founding Member — RM{price}/mo"`.
- **Locale resolution** (`strategy` in `vite.config.ts`):
  `["cookie", "preferredLanguage", "baseLocale"]` — the `PARAGLIDE_LOCALE`
  cookie wins, else the browser's preferred language, else English.
- **Switcher:** the landing `Nav` (`src/components/landing/nav.tsx`) has a
  globe toggle calling `setLocale()` (writes the cookie + reloads). Every public
  landing page renders `<Nav />`, so the toggle is available on all of them.

## Scope (what we translate)

Translation covers the **public, pre-sign-in marketing site**:

- Landing page (`/`) and all its sections (`src/components/landing/*`)
- Pricing (`/pricing`)
- Cost calculator (`/cost`)
- Buyer order tracking (`/track/<token>`)

**Out of scope** (deliberately, for now): sign-in / sign-up / onboarding, the
legal pages, the seller dashboard (`/app/*`), and the buyer storefront
(`/<slug>` + checkout). The seller-dashboard Settings & Billing tabs happen to
be translated as a side effect of recent work; the rest of the dashboard and the
storefront are tracked separately.

> SEO `<head>` meta (title/description/OG) is intentionally left in English
> across all routes for consistency — translate it only if we decide to target
> BM search separately.

## Gotcha: resolve `m.*` at render time, not module scope

`m.foo()` reads the current request's locale when called. On the SSR server,
modules are imported **once** and reused across requests, so calling `m.*` at
module top-level bakes in whichever locale was active at first import and serves
it to everyone. Always call message functions **inside** a component render (or a
hook), never in a module-level `const`. See the `useTiers`/`useFeatures` helpers
in `src/routes/pricing.tsx` for the pattern.

## Adding or changing a string

1. Add the key to **both** `messages/en.json` and `messages/ms.json` (same key,
   same `{placeholders}`).
2. Use `m.your_key()` in the component (resolved at render — see gotcha above).
3. Recompile (auto on `pnpm dev`, or run the compile command above).
4. `pnpm test` — `src/lib/i18n.test.ts` enforces that the two catalogs have
   identical key sets, no empty values, and matching placeholders per key. It
   does **not** catch hardcoded English that never goes through `m.*` — that's a
   code-review concern. A quick sweep:
   ```sh
   # user-facing JSX string literals in an in-scope file that bypass m.*
   rg '>[A-Z][a-z]' src/routes/pricing.tsx
   ```

## Naming conventions

Keys are `snake_case`, prefixed by surface: `nav_*`, `hero_*`, `faq_*`,
`pricing_*` (landing pricing teaser), `pricingpage_*` (the full `/pricing`
route), `cost_*` (the `/cost` calculator), `footer_*`, etc.
